**版本**: v1.0
**日期**: 2025-01-24
**目的**: 分析并区分零售客户和对公客户的共有信息与特有信息，为数据库设计和系统实现提供依据

## 🔄 一、共有信息（21项）

## 📐 五、数据库设计建议

### 5.1 表结构设计策略

#### 策略一：共有信息 - 统一表结构

```
crm_customer（客户主表）
├── customer_type（客户类型：1-零售，2-对公）
├── 共有字段21项对应21个功能模块
└── 通过 customer_type 区分零售/对公
```

**优点**:
- 统一管理
- 共有业务逻辑复用
- 便于跨客户类型统计

**适用模块**:
- 客户概况、标签画像、知识图谱
- 业务概览、账户、产品、授信、签约
- 评级、贡献度、营销、需求、接触轨迹

#### 策略二：特有信息 - 独立扩展表

```
零售客户扩展：
├── crm_retail_customer（零售客户扩展表）
├── crm_retail_certificate（证件信息）
├── crm_retail_family（家庭信息）
├── crm_retail_preference（客户偏好）
├── crm_customer_points（积分权益）
└── ...

对公客户扩展：
├── crm_company_customer（对公客户扩展表）
├── crm_company_organization（组织架构）
├── crm_company_contact（联系人信息）
├── crm_company_finance（财务信息）
├── crm_company_external（外部数据）
└── ...
```

**优点**:
- 清晰分离
- 易于维护
- 支持各自特有字段扩展

### 5.2 索引设计要点

| 表类型 | 关键索引 |
|--------|---------|
| **共有信息表** | `customer_id` + `customer_type` + `tenant_id` |
| **零售客户扩展表** | `customer_id`（唯一） + `tenant_id` |
| **对公客户扩展表** | `customer_id`（唯一） + `tenant_id` |
| **外部数据表** | `customer_id` + `cache_time` + `expire_time` |

### 5.3 分区表建议

| 表名 | 分区字段 | 分区策略 | 原因 |
|-----|---------|---------|------|
| **crm_customer_transaction** | transaction_time | 按月分区 | 零售+对公交易量大 |
| **crm_customer_online_behavior** | behavior_time | 按月分区 | 零售客户行为数据量大 |
| **crm_customer_statistics_snapshot** | snapshot_time | 按月分区 | 统计快照定期归档 |

## 🎯 六、实施建议

### 6.1 开发优先级

#### 第一阶段：共有信息（核心）
1. ✅ 客户主表、基础信息
2. ✅ 账户、产品、授信、签约、担保
3. ✅ 评级、贡献度、信用
4. ✅ 营销、需求、接触轨迹

#### 第二阶段：特有信息（扩展）
1. ✅ 零售客户扩展（证件、家庭、偏好、积分）
2. ✅ 对公客户扩展（组织架构、联系人、财务）

#### 第三阶段：增值服务（优化）
1. ✅ 标签画像、知识图谱
2. ✅ 线上行为、重要事件
3. ✅ 对公外部数据（工商、股东、风险、供应链）

### 6.2 数据同步策略

| 数据类型 | 同步方式 | 频率 | 说明 |
|---------|---------|------|------|
| **共有核心数据** | 实时同步 | T+0 | 账户、交易、授信等 |
| **共有评价数据** | 准实时计算 | T+1日终 | 评级、贡献度 |
| **零售权益数据** | 实时同步 | T+0 | 积分、卡券变更 |
| **对公外部数据** | 定期更新 | T+7（周） | 工商、股东、风险等 |
| **行为数据** | 准实时采集 | 小时级 | 线上渠道行为 |

### 6.3 权限控制

| 客户类型 | 权限特点 |
|---------|---------|
| **零售客户** | - 客户经理只能查看归属客户<br>- 潜在客户信息相互屏蔽<br>- 敏感信息加密（证件号、手机号） |
| **对公客户** | - 支持主协办机构/客户经理<br>- 联系人信息按权限屏蔽<br>- 财务信息按角色授权<br>- 集团客户支持层级授权 |

## 📝 总结

### 关键发现

1. **共有信息占主导**: 21项共有信息（50%）是零售和对公客户的共同基础
2. **对公客户信息更复杂**: 对公客户特有信息16项，主要是企业经营相关和外部数据
3. **零售客户更注重权益**: 零售客户特有信息8项，侧重个人偏好和权益管理
4. **外部数据依赖差异大**: 对公客户高度依赖外部数据（工商、股东、风险等），零售客户依赖较少

### 设计原则

1. ✅ **共性抽象**: 共有信息使用统一表结构，通过 `customer_type` 区分
2. ✅ **个性扩展**: 特有信息使用独立扩展表，便于维护和扩展
3. ✅ **分区优化**: 大数据量表（交易、行为、统计）使用分区表
4. ✅ **权限隔离**: 通过多租户 + 数据权限实现客户数据隔离
5. ✅ **外部缓存**: 对公外部数据采用混合缓存策略（本地缓存 + 定期刷新）
